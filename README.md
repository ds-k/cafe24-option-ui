본 프로젝트는 Cafe24 상품 상세페이지의 기본 구매 흐름을 유지하면서 커스텀 골라담기(옵션) UI를 구현한 결과물입니다.

### 🛠 기술 스택 선정 이유: Vanilla TypeScript + Vite

커스텀 UI 컴포넌트를 설계함에 있어 React 대신 **Vanilla TypeScript**를 채택했습니다.

- **DOM 충돌 방지:** Cafe24의 장바구니/구매 로직은 jQuery와 Vanilla JS 기반의 직접적인 DOM 제어로 이루어집니다. React의 Virtual DOM을 사용할 경우 네이티브 스크립트와의 타이밍 이슈나 상태 불일치가 발생할 위험이 크기 때문에, DOM을 직접 제어하면서도 타입 안정성을 챙길 수 있는 Vanilla TS를 선택했습니다.
- **번들링 최적화:** 과제 요구사항 상 스킨 HTML 내에 스크립트를 삽입해야 합니다. Vite 설정을 통해 별도의 vendor chunk 없이 단 하나의 `index.js`, `index.css` 파일로 깔끔하게 떨어지도록 빌드 환경을 구축하여 Cafe24 스킨 이식성을 극대화했습니다.

## 2. 옵션 UI 및 컴포넌트 구현 방식

- **컴포넌트 분리와 상태 관리:** 유지보수성을 위해 모든 UI 요소(`OptionCard`, `FlavorPanel` 등)를 클래스 기반 컴포넌트로 분리하였고, 이들 간의 데이터(선택된 옵션, 총 수량, 장바구니 담김 상태)는 단일 소스인 `store.ts`를 구독(Subscribe)하여 리렌더링하도록 옵저버 패턴을 적용했습니다.
- **PC/MO 환경 분리 (`PcApp.ts`, `MobileApp.ts`):** 단순히 CSS 미디어 쿼리에만 의존하지 않고, 데스크탑과 모바일의 레이아웃(특히 모바일의 고정 바텀시트 제어) 동작이 완전히 다르기 때문에 렌더러와 이벤트 리스너를 논리적으로 분리했습니다.

### 💡 구매 제한 및 Suffix(`_1`, `_2`) 관련 해석 및 구현

과제 명세의 **"옵션값의 `_1`, `_2` suffix는 같은 개입수의 추가 가능 횟수를 의미합니다"** 라는 조건에 가장 집중했습니다.

1. **설계 의도 파악:** `10개입_1`과 `10개입_2`는 서로 다른 맛을 담기 위한 슬롯이 아니라, Cafe24에서 "동일 옵션은 별도의 줄(Row)로 추가되지 않는다"는 한계를 우회하기 위해 **수량(Quantity) 자체를 옵션명으로 하드코딩한 구조**로 해석했습니다. (관리자 페이지에서 `_2`의 가격이 정확히 2배로 책정된 것이 그 증거입니다.)
2. **동기화 로직 구현:** 피그마 시안에 맞추어 유저 경험을 위한 맛 선택 기능은 구현하였으나, Cafe24와의 네이티브 연동 시에는 유저가 고른 총 수량(Total Quantity)을 합산하여 그 숫자에 정확히 매칭되는 Suffix 옵션 하나만 클릭하도록 구현했습니다.
   _(예: 10개입을 2번 고르면, 기존 `10개입_1` 옵션을 지우고 `10개입_2`를 클릭하여 수량과 가격이 연동되도록 설계)_
3. **한도 초과 방어 로직:** `cafe24Sync.ts`에서 최초 로드 시 네이티브 DOM을 파싱해 각 개입수별 Suffix의 최댓값을 `Limit`로 저장해 둡니다. 유저가 총 수량을 올릴 때 이 Limit를 초과하려 하면 조작을 차단하고 Toast 메시지를 띄워 안내합니다.

## 3. 외부 설정 JS 구조 (`config.ts`) 가이드

개발자가 아닌 작업자(MD, 디자이너 등)도 코드를 깊게 수정할 필요 없이 텍스트와 뱃지만 손쉽게 관리할 수 있도록 `config.ts` 파일 하나로 모든 데이터를 응집시켰습니다.

```typescript
// src/config.ts 의 옵션 설정부 가이드
export const OPTIONS_DATA: OptionConfig[] = [
  {
    id: "10개입", // [주의] Cafe24 원본 DOM의 옵션명(텍스트)과 정확히 일치해야 합니다.
    name: "10개입", // 유저 UI에 노출될 타이틀
    price: 24700, // UI에 표시될 기준 금액
    unitPriceText: "1개 : 2,470원", // (선택) 보조 설명 문구
    badge: "가장 많이 선택해요!", // (선택) 강조 뱃지 텍스트 (비우면 노출 안 됨)
    badgeType: "gray", // 뱃지 색상 타입 ("gray" | "red")
    count: 10, // 선택해야 할 총 맛(Flavor)의 갯수
  },
  // 동일한 양식으로 새로운 옵션 추가/삭제 가능
];
```

이 파일만 수정하면 UI 전체 컴포넌트에 데이터가 자동으로 전파되어 렌더링 됩니다.

## 4. 카페24 옵션과 동기화한 방식

Cafe24의 기본 요소를 CSS로 가리고(`display: none;`), 커스텀 UI의 액션을 원본 DOM 이벤트로 매핑하는 브릿지 역할을 수행합니다.

- **옵션 클릭 동기화:** `triggerCafe24Click(optionId, targetSuffix)`를 통해 원본 `.ec-product-button` 내의 특정 `a` 태그를 찾아 강제 `click()` 이벤트를 발생시킵니다.
- **삭제 동기화:** 커스텀 UI에서 [X] 버튼 클릭 시, 네이티브 장바구니(`totalProducts`)를 순회하여 해당 아이템의 `.option_box_del` 삭제 버튼을 찾아 이벤트를 전파합니다.
- **구매 액션 동기화:** 모바일 바텀시트에 자체 구현한 `장바구니`, `구매하기` 버튼을 누를 시 숨겨진 네이티브 `#actionCart`와 `#actionBuy` 버튼의 이벤트를 호출하여 Cafe24의 결제 플로우에 완벽하게 태웁니다.

## 5. 추가 개선 아이디어 (Further Improvements)

과제 요구사항에 집중하여 개발을 진행하면서, 추후 실무 서비스 환경으로 확장할 때 도입해 볼 만한 두 가지 아이디어를 고민해 보았습니다.

### ① 맛(Flavor) 정보 데이터 연동

현재 과제 명세에서는 Cafe24의 "옵션 추가 가능 횟수 제한" 우회 로직을 구현하는 데 집중하였기에, 사용자가 선택한 **'맛(Flavor)' 정보**는 네이티브 장바구니로 넘어가지 않고 UI 단에서만 휘발되도록 설계되었습니다.
하지만 만약 Cafe24 설정에서 텍스트를 입력받을 수 있는 옵션(예: 입력형 옵션 등)을 지원한다면, 현재 설계된 `store.ts`에서 조합된 맛 텍스트(예: _"떡볶이 5개, 마라 5개"_)를 직렬화하여 해당 Input 폼에 자동 삽입(Value Injection)하는 방식으로 쉽게 주문서 연동이 가능할 것이라고 판단됩니다.

### ② 외부 설정(`config.ts`)의 Database화 (CMS 연동)

과제에 맞추어 `config.ts` 파일 하나만 수정하면 디자인과 텍스트를 바꿀 수 있도록 구현하였으나, 여전히 비개발자(기획자, 디자이너)가 코드 저장소에 접근하여 파일을 수정해야 하는 허들이 있습니다.
이를 개선하기 위해, 기획자들이 친숙하게 사용하는 **Notion API**를 활용하거나 간단한 백엔드 DB와 Admin 페이지를 연동한다면, 비개발자도 손쉽게 뱃지 문구나 옵션 구성을 변경할 수 있는 완벽한 사내 관리 시스템(CMS)으로 고도화할 수 있을 것입니다.

## 6. 별도 신경 쓴 사항 (최적화)

- **이미지 Base64 인라이닝 최적화:** 맛을 표현하기 위한 썸네일 이미지들을 별도의 스토리지나 S3에 올리지 않았습니다. 대신 작은 용량의 이미지들은 Vite 빌드 파이프라인에서 자동으로 **Base64 String**으로 인코딩하여 `index.js`에 포함되도록 설정(`assetsInlineLimit` 조정)했습니다. 이를 통해 타 쇼핑몰 스킨에 붙여 넣을 때도 CORS 에러나 이미지 엑스박스 이슈 없이 단일 JS 삽입만으로 완벽하게 동작합니다.
